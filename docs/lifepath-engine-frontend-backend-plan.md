# LifePath Engine
## 调一下参数，你的人生会走向哪里？

## 1. 文档定位

本文档定义 `LifePath Engine` 的前后端规划，目标不是写到底层实现细节，而是明确：

- 前端需要承载哪些页面和交互
- 后端需要承担哪些能力和模块
- 前后端如何配合把“人生路径推演”做成可运行产品

## 2. 总体架构思路

系统建议分成四层：

- 前端交互层：负责建模、图谱查看、路径浏览、节点修改、建议查看
- 后端应用层：负责项目管理、参数处理、任务编排、结果输出
- 推演引擎层：负责人生图谱、智能体行动、状态变化、分支生成、记忆回写
- 存储层：负责用户档案、路径版本、图谱数据、推演 checkpoint、历史记录

**MVP 实际采用**：`前端 Vite + Vanilla JS（SPA）+ 后端 FastAPI + 本地 JSON 文件存储`，无需数据库。

## 3. 前端规划

### 3.1 前端目标

前端不是单纯展示结果，而是承载一个完整的人生推演工作台。

它要支持：

- 从零建立用户背景
- 查看系统如何从关注参数发散出图谱和智能体
- 观看路径生成过程
- 深入查看单条路径的节点逻辑
- 回到某个节点修改条件并重跑
- 接收 AI 给出的行动建议

### 3.2 前端技术建议

- `Vite` 作为开发/构建工具
- `Vanilla JavaScript`（ES Module）— 无框架依赖
- 自定义 CSS 设计系统（基于 Stitch "Forensic Expert" 风格）
- SVG 内联渲染路径树图谱
- `SSE`（Server-Sent Events）用于推演过程实时反馈
- **数据存储**：不使用数据库，所有数据以 JSON 文件形式存储在 `backend/data/projects/` 目录下

结果：

#### 4. 上传建模页

#### 6. 图谱生成页

作用：

- 用户输入与系统扩展的来源标记

图谱层建议：

- 直接移植或复用 MiroFish 的图谱交互与动效
- 节点点击展开详情
- 边关系动态高亮

#### 7. 推演过程页

#### 8. 路径总览页

作用：

每条路径卡片建议展示：

建议内容：

## 4. 后端规划

### 4.1 后端目标

后端不是普通 CRUD 服务，它是这个产品的“推演操作系统”。

后端负责：

- 接收用户资料和参数
- 结构化建模
- 发散影响因素
- 生成人生图谱
- 动态生成智能体
- 多轮推演
- 分支管理
- 节点回退重跑
- 输出 AI 建议

### 4.2 后端技术建议

推荐第一选择：

- `FastAPI`
- `Pydantic`
- **本地 JSON 文件存储**（每个项目一个 JSON 文件，`backend/data/projects/{id}.json`）
- `OpenAI SDK`（预留，LLM 推理调用）

原因：

- `FastAPI` 更适合异步任务和流式进度反馈
- `Pydantic` 便于定义复杂的节点和状态模型
- **无需安装数据库**，JSON 文件存储足以满足 MVP 阶段需求
- 每个项目自包含一个 JSON 文件，便于调试、导出和分享，后续可迁移到 PostgreSQL

### 4.3 后端模块划分

#### 1. 项目管理模块

负责：

- 创建推演项目
- 保存用户档案
- 保存关注参数
- 查询历史项目

#### 2. 用户建模模块

负责：

- 问卷答案转结构化画像
- 上传材料解析
- 提取用户背景标签
- 生成基础人生 Profile

#### 3. 参数扩展模块

负责：

- 分析用户关注参数
- 自动发散多层影响因素
- 形成因果因素树
- 识别本次推演需要哪些智能体

#### 4. 人生图谱模块

负责：

- 生成初始人生图谱
- 创建节点和关系
- 为推演引擎提供上下文

#### 5. 智能体生成模块

负责：

- 生成核心智能体
- 绑定智能体的立场、资源、偏好和影响范围
- 按问题动态增删智能体

#### 6. 推演引擎模块

负责：

- 多轮推进
- 智能体行动
- 事件产生
- 状态更新
- 分支生成
- 分支剪枝
- 阶段总结

#### 7. 记忆与回写模块

负责：

- 把每轮结果写回图谱
- 保存节点历史
- 保存状态变化
- 为下一轮提供上下文

#### 8. 节点重跑模块

负责：

- 定位路径中的某个节点
- 生成 checkpoint
- 从该节点恢复上下文
- 应用新条件后局部重跑

#### 9. AI 建议模块

负责：

- 满意路径的行动建议
- 不满意路径的应对建议
- 差距分析
- 下一步计划

## 5. 推演引擎规划

### 5.1 核心运行单位

建议把推演单位设计成“人生节点”。

节点结构建议包含：

- `node_id`
- `node_type`
- `title`
- `description`
- `time_index`
- `trigger_reason`
- `influencing_factors`
- `affected_states`
- `branch_id`
- `editable_fields`

### 5.2 节点类型

- `decision`
- `opportunity`
- `result`
- `cascade`
- `risk`
- `reflection`

### 5.3 状态模型

后端需要维护一个阶段性人生状态对象，例如：

- 学业状态
- 职业状态
- 收入状态
- 城市状态
- 关系状态
- 家庭压力
- 心理压力
- 健康状态
- 可选择空间

建议把状态更新设计成：

- 一部分由规则计算
- 一部分由 LLM 生成解释和事件内容

这样可控性更高，避免推演完全漂移。

### 5.4 回合机制

每轮执行顺序建议为：

1. 读取当前分支状态和记忆
2. 让相关智能体提出行动或影响
3. 生成新节点
4. 更新人生状态
5. 写回图谱和记忆
6. 判断是否生成分支
7. 输出阶段总结

### 5.5 分支策略

为了避免分支爆炸，建议控制：

- 每轮最多扩展的候选分支数
- 相似分支合并
- 低价值分支剪枝
- 保留代表性分支

代表性分支建议包括：

- 最优解倾向
- 保守解倾向
- 风险解倾向
- 平衡解倾向

## 6. 节点回退与重跑设计

### 6.1 为什么要单独设计

这是产品的关键差异点。

普通推演系统只会给用户一个结果，但 `LifePath Engine` 必须允许用户说：

- 这条路我喜欢
- 但这里我不接受
- 把这个节点改掉，再继续跑

### 6.2 实现逻辑

后端需要为关键节点保存 checkpoint，至少包括：

- 当时的分支状态
- 当时的图谱快照
- 当时的智能体记忆摘要
- 当时的用户约束

当用户修改节点后：

- 系统复制当前分支为新分支
- 载入目标节点的 checkpoint
- 应用修改
- 从该节点继续推演

### 6.3 前后端配合

前端：

- 在路径详情页选择节点
- 弹出编辑面板
- 提交修改后的值

后端：

- 校验节点可编辑性
- 创建新分支版本
- 启动局部重跑任务
- 返回新路径 ID

## 7. AI 建议模块规划

### 7.1 输入

建议模块的输入包括：

- 用户当前背景
- 当前选择的路径
- 路径中的关键节点
- 用户满意或不满意的反馈
- 当前现实可执行条件

### 7.2 输出

#### 满意路径模式

输出：

- 路径成真的前提条件
- 必须提前准备的资源
- 最关键的三个行动点
- 未来 1 个月 / 3 个月 / 1 年建议

#### 不满意路径模式

输出：

- 当前风险成因
- 最值得干预的节点
- 哪些条件可逆
- 哪些替代路径更现实
- 如何降低损失并争取更优发展

## 8. 数据存储规划

### 8.1 建议存储对象

- `projects`
- `user_profiles`
- `concern_parameters`
- `agents`
- `graphs`
- `branches`
- `nodes`
- `state_snapshots`
- `checkpoints`
- `advice_results`

### 8.2 为什么先用 PostgreSQL

因为第一版最重要的是：

- 项目和版本管理
- 分支和节点查询
- checkpoint 回溯
- 结果对比

这些数据更像结构化业务数据，不一定需要先上图数据库。

图谱遍历如果前期需求不复杂，可以用：

- 关系表
- JSONB
- 缓存后的前端图数据

### 8.3 图数据库策略

如果后续图遍历和关系查询复杂度明显上升，再考虑：

- `Neo4j`
- 或继续保留 Postgres，增加图查询层

## 10. 开发顺序建议

### 阶段一：MVP

- 建模入口
- 参数输入
- 参数扩展
- 初始图谱生成
- 3 条路径推演
- 路径结果页
- 节点回退重跑
- AI 建议

### 阶段二：增强版

- 更丰富的智能体
- 更细粒度状态机
- 更复杂的分支剪枝
- 更完整的历史项目管理
- 更长时间跨度推演

### 阶段三：高级版

- 多轮深度对话式校准
- 更强的现实数据接入
- 多用户协作与分享
- 报告导出

## 11. 对 MiroFish 的复用策略

建议直接复用或移植的部分：

- 图谱可视化层
- 节点动效和布局思路
- 路径扩展的表现方式
- 实时进度反馈的交互节奏

不建议直接照搬的部分：

- 社交传播语义
- OASIS 社交动作模型
- Twitter / Reddit 平台抽象

要重写的核心是：

- 节点语义
- 智能体语义
- 状态变化模型
- 分支策略
- 节点回退重跑机制

## 12. 风险提示

- 如果直接复用 MiroFish 前端代码，需要确认许可证兼容性和引用边界。
- 如果推演完全依赖 LLM 自由生成，结果会不稳定，必须加入规则层和状态机。
- 节点回退功能会显著提高后端复杂度，建议从一开始就设计 checkpoint 体系。

## 13. 最终结论

`LifePath Engine` 的前端应做成“人生路径工作台”，后端应做成“人生推演引擎 + 分支管理系统”。

最重要的不是把页面做全，而是把以下闭环先跑通：

`建模 -> 参数扩展 -> 智能体生成 -> 人生图谱 -> 多路径推演 -> 节点回退重跑 -> AI 建议`

这个闭环一旦跑通，产品就成立了。
