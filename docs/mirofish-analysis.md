# MiroFish 推演技术文档

## 1. 文档目的

本文基于 MiroFish 公开资料与本地参考源码，对它的推演机制做技术拆解，目标不是复述宣传语，而是回答两个问题：

1. MiroFish 到底如何把一份“种子材料”变成可运行的推演世界。
2. 它的哪些机制可以迁移到“人生未来路径推演”项目。

说明：

- 已由公开文档确认的内容，主要来自 README、官网描述与仓库源码。
- 对运行机制的抽象与迁移建议，属于基于源码结构做出的工程推断。

## 2. 结论先行

MiroFish 不是一个“直接让 LLM 给答案”的预测器，而是一条分层流水线：

1. 把现实材料转成可计算的实体关系图。
2. 把图中的关键实体转成可行动的 Agent。
3. 用 LLM 为 Agent 补人格、活跃度、立场、影响力、节奏。
4. 在两个社交平台环境中并行运行多轮行为模拟。
5. 把模拟中产生的新行为继续写回图谱，形成时序记忆。
6. 最后再由 ReportAgent 基于图谱、日志、访谈结果生成解释性报告。

换句话说，MiroFish 的“推演”不是一次生成，而是：

`文本种子 -> 图谱建模 -> Agent 化 -> 多轮交互 -> 记忆回写 -> 报告总结`

这也是它最值得照搬的部分。

## 4. MiroFish 的技术分层

### 4.1 输入层

输入不是结构化数据库，而是两类自然语言信息：

- 种子材料，例如新闻、报告、小说、事件描述。
- 推演要求，例如“预测舆情如何发展”“某事件的未来走向”。

这一步的价值不在于直接回答，而在于让系统先得到“世界底稿”。

### 4.2 本体层

`ontology_generator.py` 会先把文本分析成适合社交仿真的本体：

- 生成实体类型，且要求是“能发声的主体”，如人、机构、媒体、学校、政府部门。
- 生成关系类型，如隶属、回应、支持、反对、报道、合作。
- 强制保留兜底类型 `Person` 和 `Organization`。

这里的本质是先决定“世界里允许出现哪些角色、角色之间允许有什么关系”。

### 4.3 图谱层

`graph_builder.py` 使用 Zep：

- 先创建图谱。
- 把本体写入图谱定义。
- 把原始文本切块后批量写入 Zep。
- 等待 Zep 完成图谱抽取。

最终得到的是一个 GraphRAG 世界模型，而不是一段摘要。

这个阶段的作用有两个：

- 给后续 Agent 生成提供结构化上下文。
- 给后续报告与检索提供长期记忆底座。

### 4.4 实体过滤层

`zep_entity_reader.py` 会从图谱里把实体读出来，并筛掉不适合做社交主体的节点。

它会保留：

- 节点类型
- 节点摘要
- 相关边
- 关联节点上下文

这一步非常关键，因为不是所有图谱节点都适合被变成 Agent。MiroFish 实际上做的是：

`图谱节点全集 -> 可发声主体集合`

### 4.5 Agent 人设层

`oasis_profile_generator.py` 与 `simulation_manager.py` 会把实体转成 OASIS 可运行的人设档案。

这里至少包含：

- 实体名称
- 实体类型
- 背景摘要
- 可能的角色定位
- 可被平台环境读取的 profile 文件

源码里同时支持：

- 规则生成
- LLM 增强生成

所以 MiroFish 的 Agent 不是从零想象出来的，而是“图谱实体的人格化投影”。

### 4.6 仿真配置层

`simulation_config_generator.py` 是核心之一，它并不只生成一个 `max_rounds`，而是生成完整的模拟参数：

- 时间配置
- 每轮对应的模拟分钟数
- 每小时活跃 Agent 数量区间
- 热点主题
- 初始事件和初始帖子
- 每个 Agent 的活跃时段
- 发帖频率、评论频率
- 响应延迟
- 情绪偏向
- 立场
- 影响力权重
- 平台参数

它本质上在做一件事：

`把抽象的推演要求，翻译成可执行的仿真规则`

也就是说，MiroFish 的“预测要求”不是直接喂给最终报告，而是先变成模拟环境参数。

### 4.7 运行层

`run_parallel_simulation.py` 显示，MiroFish 的主执行方式是双平台并行：

- Twitter 世界
- Reddit 世界

两个平台共用同一套上层配置，但使用不同的动作集合与环境。

这不是 UI 花活，而是一个重要设计：

- 同一批角色在不同传播机制里，会表现出不同的扩散路径。
- 双平台相当于并行运行两个“社会规则略有差异的世界”。

### 4.8 轮次推进层

在每轮模拟中，系统大致按下面的逻辑推进：

1. 根据总时长和 `minutes_per_round` 计算总轮数。
2. 根据当前模拟时刻，确定当前小时、天数。
3. 根据活跃时段、活跃度与全局人数阈值，挑选本轮活跃 Agent。
4. 活跃 Agent 在环境中执行动作。
5. 从数据库中读取本轮真实发生的动作。
6. 写入动作日志和回合摘要。

动作类型包括但不限于：

- 发帖
- 评论
- 点赞/踩
- 转发
- 引用
- 关注
- 搜索
- 屏蔽

因此它不是“节点树静态展开”，而是“多轮行为驱动的状态演化”。

## 5. MiroFish 真正的推演机制

如果把 MiroFish 的推演抽象成算法，它更接近下面这套模式。

### 5.1 用世界模型替代一次性回答

先生成图谱，再生成 Agent，再让 Agent 互动。

这说明 MiroFish 认为预测不是一句话答案，而是一个世界内部的演化结果。

### 5.2 用角色差异替代平均化推断

不同 Agent 有不同：

- 身份
- 立场
- 影响力
- 响应速度
- 活跃时段

这保证了结果不是“所有人都像同一个 LLM”。

### 5.3 用事件驱动替代空转

系统会生成：

- 初始帖子
- 热点话题
- 叙事方向

这相当于在仿真世界里点燃第一簇火种，然后让扩散自然发生。

### 5.4 用平台规则制造差异化传播

Twitter 和 Reddit 的动作集合、互动方式、推荐机制并不完全相同。

所以同一事件会在不同平台呈现不同节奏和分叉。

### 5.5 用记忆回写形成“演化中的世界”

`zep_graph_memory_updater.py` 会把模拟中的行为转成自然语言 episode，再批量写回 Zep 图谱。

这意味着：

- 图谱不是只在模拟前构建一次。
- 模拟后的新事实会进入长期记忆。
- 报告和访谈读取到的是“演化后世界”。

这一点很重要，它让系统从静态知识图谱变成动态时序记忆图谱。

### 5.6 用访谈与报告做结果解释

模拟结束后，环境不会立刻关闭，而是进入等待命令模式，支持：

- 采访单个 Agent
- 批量采访多个 Agent
- 采访全部 Agent

接着 `report_agent.py` 用 ReAct 风格生成报告：

- 先规划目录
- 再逐章调用工具
- 检索图谱
- 汇总模拟结果
- 生成完整说明

所以 MiroFish 的输出不是只有“结论”，而是同时给出：

- 结果
- 路径
- 证据
- 角色视角

## 6. 用一句话概括它是怎么推演的

MiroFish 的推演方式可以概括为：

`把文本中的现实世界先结构化为图谱，再人格化为 Agent，再在可控规则下多轮互动，让新行为继续改写世界，最后由报告代理提炼未来走向。`

## 7. 对“节点不断延展”的理解

你提到想“照抄 MiroFish，把每个节点不断延展”。从源码看，MiroFish 实际并不是前端上直接递归展开一棵树，而是通过下面三个层面实现“延展”：

1. 图谱延展  
   初始材料先扩成实体关系网络。

2. 行为延展  
   每轮每个 Agent 的行为会触发新的帖子、评论、关注、转发等。

3. 记忆延展  
   新行为继续回写图谱，成为下一阶段上下文。

所以你如果做人生推演，不该只抄“树状 UI”，更应该抄这套三段式：

`因果扩展 -> 行为扩展 -> 记忆扩展`

## 8. 可迁移到人生推演项目的核心机制

### 8.1 可直接迁移

- 先建世界模型，再做推演。
- 先抽实体，再做 Agent 化。
- 先生成规则，再跑多轮模拟。
- 允许用户中途注入变量。
- 保留可采访的角色与可解释的报告。
- 把模拟产生的新事实写回长期记忆。

### 8.2 需要改造

MiroFish 面向的是“舆论传播”，你要做的是“人生路径演化”，因此以下对象要被替换：

| MiroFish 对象 | 人生项目中的对应物 |
| --- | --- |
| 舆情事件 | 人生抉择/人生阶段 |
| 社交平台环境 | 人生阶段环境 |
| 发帖/评论/转发 | 决策/反馈/机会/代价 |
| 舆论主体 | 个人、家人、学校、导师、公司、伴侣、朋友、政策、城市 |
| 热点话题 | 关键变量与冲突焦点 |
| 平台传播差异 | 不同人生路线规则差异 |

## 9. 对你项目最重要的启发

你要复制的不是“舆情模拟”这个题材，而是这套生产链：

1. 用户输入少量关键参数。
2. 系统自动扩展相关人物、机构、环境因素。
3. 这些因素进入一张可解释的因果图/关系图。
4. 图中的角色被转成多种 Agent。
5. Agent 在按年、按季度或按关键节点推进的环境中互动。
6. 每个阶段产生新的机会、风险、关系变化、资源变化。
7. 这些变化继续写回世界状态。
8. 最终输出多条未来路径，而不是单一答案。

## 10. 对人生推演项目的技术迁移建议

建议你保留 MiroFish 的四个骨架：

### 10.1 GraphRAG 骨架

把“人生基础信息 + 目标 + 关键抉择 + 外部环境”转成一张人生因果图。

### 10.2 Agent 骨架

不是所有 Agent 都是“人”，还可以是：

- 学校系统
- 导师
- 公司
- 城市机会结构
- 家庭约束
- 财务压力
- 健康状态

### 10.3 Round-based Simulation 骨架

每轮不一定是一小时，可以改成：

- 一个月
- 一学期
- 一年
- 一个关键人生节点

### 10.4 Report + Interview 骨架

最终不仅给路径，还允许用户追问：

- 为什么这条路会变成这样？
- 五年后这个“我”怎么看当年的选择？
- 导师 Agent 为什么在第三年不再支持我？

## 11. 风险提醒

把 MiroFish 迁移到人生推演时，有一个本质风险：

舆情模拟偏向群体传播，人生推演偏向个体命运。后者更容易被用户误认为“算命”或“真实预测”。

所以你的产品必须明确：

- 它是基于参数与行为规则的情景推演系统。
- 它给出的是可能路径，不是确定命运。
- 它更适合辅助决策，不适合给用户制造“宿命感”。

## 12. 最终抽象

如果只保留 MiroFish 最核心的一句话方法论，可以写成：

`先把世界建出来，再让角色活起来，再让时间往前走。`

你的项目只需要把“舆论世界”替换成“人生世界”。
